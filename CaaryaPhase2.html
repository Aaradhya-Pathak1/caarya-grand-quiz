01 · Work Unit Definition

What I was actually trying to do

I built a single-file web application: a 12-question, three-round trivia quiz that records each player's score and completion time to a Firebase Realtime Database, then renders a live analytics dashboard benchmarking that player against every prior participant: a percentile, a cognitive bracket, a bell curve, and a score-projection matrix.

The Atomic Values I was creating: (1) make self-assessment frictionless by showing results instantly and visually; (2) produce a percentile signal that stays trustworthy even when only a handful of people have played; and (3) turn raw right/wrong answers into an interpretable performance narrative.

Almost none of my real learning came from the parts that worked the first time. It came from a chain of failures where the code threw no error, the screen looked fine, and the behaviour or the stored data was silently wrong. Every card below is one of those.

02 · Knowledge Cards

Seven insights from the build

01A change you can't see persisted is not a change
Atomic Value
Reduced silent failures when editing files programmatically.
Situation / Context
Expanding the quiz from 6 to 12 questions. I was using scripts to edit the HTML file's logic in bulk rather than by hand.
The Friction
Round 1's answer buttons went completely dead. No error popup, no console crash visible to the user, clicking simply did nothing.
What Failed
I checked the obvious suspects first: the button click handlers, the function that builds the options, the event binding. All correct. I even ran a syntax check, it passed. None of that was the problem, and I lost time there.
What Worked
I stopped checking the logic and inspected the actual values in the saved file. The total was still 6; the points table and round-3 data were undefined. Root cause: the edit script changed the file in memory and printed a "success" check, but never wrote the result back to disk. Every later script then re-loaded the original, unchanged file.
The Principle
Verify the side effect, not the intention. A "✓ done" log on an in-memory variable proves nothing, the only proof is reading the persisted artifact back. The bug wasn't in the code that runs; it was in the code that saves.
When This Applies
Any scripted file edit, database migration, or batch transform with a write step. Irrelevant for pure functions that return a value and touch nothing external.
AI Prompt Used
"the R1 quiz buttons are unresponsive"
Deceptively simple. The fix was nowhere near the buttons.
02Change a data shape, and every reader breaks silently
Atomic Value
Eliminated silent data-shape mismatches across the app.
Situation / Context
Round 3 introduced a "skipped" state (timer ran out). I changed every result from {correct: true} to {result: 'correct' | 'wrong' | 'skipped'} to support it.
The Friction
The cumulative-score trajectory chart drew a dead-flat line along the bottom, every point pinned at zero.
What Failed
I first suspected the chart's coordinate math and y-axis scaling. That was a red herring.
What Worked
The chart's counter still read the old field: if (r.correct) cum++. That property no longer existed, so it returned undefined every time, the counter never moved, and every point sat at zero. Changed it to r.result === 'correct'.
The Principle
In JavaScript, reading a property that doesn't exist returns undefined; it does not throw. So a schema change creates an invisible bug in every consumer you forget to migrate. The real fix is never just the write site; it's tracing every read.
When This Applies
Any data-model change in a loosely-typed language. A typed language (TypeScript) would have flagged this at compile time, which is itself the takeaway.
AI Prompt Used
"the cumulative score trajectory ... is not good, correct them"
03A spec value hardcoded in two places will drift apart
Atomic Value
Stopped permanent data corruption at the storage boundary.
Situation / Context
The pipeline that writes each player's score to Firebase, after the quiz grew from 6 to 12 questions.
The Friction
Nearly invisible, no crash, and scores looked correct on the results screen. The damage was happening in the database, not the UI.
What Failed
I'd updated the total to 12 and the display logic, and assumed the data-writing layer had followed along.
What Worked
A verification question forced me to re-read the write code, where I found Math.min(6, score), a leftover from the 6-question era. Every score of 7 to 12 was being silently clamped to 6 before being stored, quietly poisoning the entire global distribution. Changed to clamp at 12.
The Principle
A number that encodes a spec value (the max score) must come from a single source of truth, not be retyped wherever it's needed. The most dangerous bugs don't crash; they produce plausible-looking wrong data, and at a storage boundary that corruption is permanent.
When This Applies
Anywhere a constant appears more than once, especially at write-to-storage points. Less critical for display-only values you can recompute.
AI Prompt Used
"you are calculating percentile based on scores first and then time right?"
A sanity-check question, not a bug report. It surfaced the bug anyway.
04A correct number from too little data is still a lie
Atomic Value
Made a statistic trustworthy at small sample sizes.
Situation / Context
Computing a global percentile from the live distribution of every player's score in Firebase.
The Friction
A player who got 7/12 was shown at the 95th percentile. It felt obviously wrong for a middling score.
What Failed
My instinct was "the math is broken." It wasn't. With only ~3 people in the database and this player being the fastest, they had genuinely beaten 95% of that tiny pool. The calculation was correct and the result was meaningless.
What Worked
A Bayesian blend: a hand-built theoretical percentile table acts as a prior, and the live data's influence grows with sample size, blend = min(1, players / 100). At 3 players the result hugs the theory; at 100+ it trusts the live data fully.
The Principle
Statistical confidence scales with sample size. A "correct" percentile from n=3 is just noise dressed up as a real number. Leaning on a sensible default when data is thin, and trusting the live numbers only as they accumulate, keeps the result honest. This is the basic idea behind Bayesian shrinkage.
When This Applies
Any rating, ranking, or average over a small or growing population, early product reviews, new leaderboards, A/B tests before they reach significance. Unnecessary once samples are large and stable.
AI Prompt Used
"how is 7/12 a 95 percentile? it shouldn't be so high"
05Stitch two data sources, and their shared rules don't survive
Atomic Value
Guaranteed logical consistency in the projection display.
Situation / Context
The score-projection matrix shows what percentile each nearby score would earn. Your own row used live data; the projected rows used the static estimate table.
The Friction
Rows for higher scores were displaying lower percentiles than the user's own, e.g. 9/12 at 80th while the player at 7/12 sat at 81st. Impossible: more correct answers cannot rank you worse.
What Failed
Filling each row from its own independent source. The live scale and the static scale didn't agree, so the ordering broke.
What Worked
Anchor every projected row to the player's actual percentile plus the theoretical gain: rowPct = yourPct + (estimate[s] − estimate[yourScore]). Higher score now always maps to a higher percentile by construction.
The Principle
When you combine two data sources, an invariant each satisfies alone (here: monotonicity) is not automatically preserved in the combination. You have to name the invariant and enforce it explicitly.
When This Applies
Merging datasets, blending model outputs, combining live and cached values, any UI showing a derived ordering.
AI Prompt Used
"9/12 and 8/12 should not be lower than my score ... pls fix everything"
06Edge cases deserve a line of code, not a formula's guess
Atomic Value
Made the top-end result feel correct and earned.
Situation / Context
The Bayesian blend from Card 04 applied to every score, including a perfect 12/12.
The Friction
A perfect score showed as 98th percentile instead of the top. The blend was averaging the theoretical 99 against a slightly lower live value and dragging it down.
What Failed
Trusting the general formula to handle the boundary. A perfect score is conceptually the ceiling, but the formula treated it as just one more data point.
What Worked
An explicit early return: if (score === 12) return 99. No blending at the boundary.
The Principle
General formulas are built for the general case; boundaries (max, min, empty, single item) routinely violate their assumptions and deserve explicit, readable handling. Bonus: the 100th percentile is mathematically undefined, you can't outrank yourself, so 99 is the honest ceiling.
When This Applies
Any formula with meaningful limits, score caps, division near zero, empty collections, first/last elements.
AI Prompt Used
"shouldn't one get 100 percentile at 150 points (max)?"
07The impressive tool isn't always the right one
Atomic Value
Traded a flashy capability for reliability the product actually needed.
Situation / Context
Early on, I generated the quiz questions and their images dynamically using the Gemini API, so every play could be different.
The Friction
Live generation meant I couldn't guarantee the questions were correct, fair, or consistently difficult, and every play added network latency, an API cost, and a dependency that could fail.
What Failed
The "impressive" version. AI-generated trivia is fine as a demo, but for an assessment that ranks people, unverifiable and inconsistent questions undermine the entire point, a fair score needs a fair, fixed test.
What Worked
I replaced it with a hand-curated, fixed question bank and lightweight inline SVG images. Reliable quality, consistent difficulty, zero latency, zero cost, and it kept the project a single self-contained file with no external dependencies.
The Principle
Choose tools by what the product needs, not by what's most impressive. For content that must be correct and consistent, a controlled fixed source beats live generation. Cutting a dependency is often a feature, not a downgrade.
When This Applies
Any decision between dynamic AI generation and curated content, quizzes, assessments, anything where correctness and fairness matter more than novelty. Less relevant for purely exploratory or entertainment features.
AI Prompt Used
Initially used Gemini API calls to generate questions + images; later removed entirely in favour of a fixed bank.
